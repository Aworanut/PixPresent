FaceFind

Event Photo Distribution SaaS · Face Recognition

A SaaS platform for event organizers to distribute photos to attendees efficiently. Guests find their own photos instantly by uploading a selfie — no signup, no login required, powered by face recognition.

FaceFind คือ SaaS platform สำหรับ Event Organizer ที่ต้องการแจกรูปถ่ายให้แก่ผู้เข้าร่วมงานอย่างมีประสิทธิภาพ ระบบใช้เทคโนโลยี Face Recognition ให้แขกค้นหารูปของตัวเองได้ทันทีโดยเพียงแค่อัพโหลดเซลฟี่ ไม่ต้องสมัครสมาชิกหรือ login

1.1 ปัญหาที่แก้ไข

  • หลังงาน event ช่างภาพมีรูปหลายร้อยใบ แต่การแจกรูปให้ถูกคนทำได้ยาก
  • แขกต้องไล่ดูรูปทีละใบจากจำนวนหลักร้อยถึงพัน ใช้เวลานานและไม่สะดวก
  • Organizer ไม่มีเครื่องมือจัดการการแจกรูปแบบ scalable

1.2 Solution

  • แขกอัพโหลดเซลฟี่ → ระบบค้นหารูปที่มีใบหน้าตรงกันจากทุกรูปในงาน → แสดง Gallery ทันที
  • Organizer จัดการทุกอย่างผ่าน Dashboard เดียว ตั้งแต่ sync รูปจนถึงแชร์ link
  • รองรับช่างภาพหลายคนต่อ event ผ่าน folder แยกกัน

2.1 Organizer Journey

  1. สมัคร account และสร้าง Organization
  2. เชื่อมต่อ Google Drive หรือ Dropbox
  3. สร้าง Event และเพิ่ม folder ของ Google Drive ที่ช่างภาพจะอัพรูป (ใส่ได้หลาย folder, ติด label ให้แต่ละทีม)
  4. Share editor access ของ folder ให้ช่างภาพแต่ละทีม → พวกเขาอัพรูปเข้า folder ของ Organizer โดยตรง
  5. รอให้ช่างภาพอัพรูปครบ แล้วกด Sync & Index
  6. ตั้งค่า Blacklist (ถ้ามี) สำหรับบุคคลที่ไม่ต้องการให้ถูกค้นหา
  7. Generate Guest Link และแชร์ให้ผู้เข้าร่วมงาน

2.2 Guest Journey

  1. เปิด Guest Link ที่ได้รับ
  2. อัพโหลดรูปเซลฟี่ที่ชัดเจน (เห็นหน้าเต็ม)
  3. รอระบบประมวลผล (~2-5 วินาที)
  4. ดู Gallery รูปทั้งหมดที่ตัวเองอยู่
  5. Download รูปแบบ Web-optimized หรือ Full Resolution

3.1 Architecture Diagram

Flow หลักของระบบแบ่งเป็น 2 ส่วน:

Organizer Indexing Flow
  1. ช่างภาพอัพรูปเข้า Google Drive / Dropbox folder ที่กำหนด
  2. Organizer กด "Sync & Index" ใน Dashboard
  3. ระบบดึงไฟล์ → Copy ไปเก็บที่ Cloudflare R2 (ทั้ง Web-optimized และ Full Resolution)
  4. ส่งรูปไปยัง AWS Rekognition เพื่อ index ใบหน้า → บันทึก Face ID ลง Supabase
  5. Organizer Generate Guest Link → แชร์ให้แขก
Guest Search Flow
  1. Guest เปิด Link → อัพโหลดเซลฟี่
  2. ระบบส่ง selfie ไปยัง AWS Rekognition SearchFacesByImage
  3. กรอง Face ID ที่อยู่ใน Blacklist ออก
  4. Query Supabase เพื่อดึง Photo metadata ที่ match
  5. แสดง Gallery จาก Cloudflare R2 CDN URL

3.2 Rekognition Strategy

  • สร้าง 1 Face Collection ต่อ 1 Event เพื่อแยก scope ชัดเจน
  • ลบ Collection หลังงานสิ้นสุดเพื่อลด cost
  • Blacklist ทำงานที่ DB layer (filter ก่อน return) ไม่กระทบ Rekognition

5.1 — Organizer Dashboard
F-01AuthenticationP0
  • สมัคร / Login ด้วย Email + Password ผ่าน Supabase Auth
  • รองรับ Password reset
  • Session คงอยู่ข้ามการปิด browser
F-02Event Management + Multi-FolderP0
  • สร้าง Event: ชื่องาน, วันที่ (Storage provider: Google Drive เท่านั้นใน Phase 1)
  • ผูก Google Drive folders ได้หลายอัน (label + URL/ID) ผ่านตาราง event_storage_folders — รองรับงานที่มีหลายทีม (เช่น ทีมหลัก, ทีม Drone, ทีม Video)
  • Workflow: Organizer สร้าง folder กลางใน Drive ของตัวเอง → share editor access ให้ช่างภาพแต่ละทีม → พวกเขาอัพรูปเข้ามาตรง (ไม่ต้องมี Drive ของตัวเอง)
  • ดูรายการ Event ทั้งหมด พร้อมสถานะ (not synced / synced / link active)
  • ลบ Event (soft delete) พร้อม cleanup Rekognition Collection
  • Note ไม่มี photographers table แล้ว — attribution ของช่างภาพมาจาก EXIF metadata หรือกรอกเองตอน commerce report (Phase 2)
F-04Sync & IndexP0
  • ปุ่ม "Sync & Index" ใน Event detail page
  • OAuth Flow: Organizer เชื่อมต่อ Google account ครั้งเดียว — ทุก folder ใน event_storage_folders ต้องอยู่ใน Drive ของ Organizer หรือ shared editor
  • ระบบ scan ทุก folder ของ event (recursive ถ้าจำเป็น) → ดึงรูปใหม่ที่ยังไม่ได้ index
  • ใช้ Batch Processing + Exponential Backoff รองรับ Google Drive API Rate Limit (HTTP 429) — ไม่ block UI ระหว่าง sync
  • แสดง progress bar ระหว่าง sync (จำนวนรูปที่ประมวลผลแล้ว / ทั้งหมด + folder ปัจจุบัน)
  • Copy รูปไป R2 ทั้ง web-optimized (JPEG 85%, max 1920px) และ full resolution
  • ส่งไปยัง Rekognition เพื่อ detect และ index faces
  • บันทึก rekognition_face_ids[] ลง photos table
  • รองรับการ re-sync เมื่อช่างภาพเพิ่มรูปเพิ่มเติม
F-05Blacklist ManagerP0
  • หลัง sync เสร็จ Organizer เลือกดูรูปทีละใบใน dashboard
  • ระบบแสดง bounding box รอบใบหน้าแต่ละคนในรูป
  • คลิกที่ใบหน้าเพื่อ "บล็อก" → เพิ่ม Face ID ลง face_blacklist
  • ดูและจัดการรายการ blacklist ทั้งหมดของ event
  • ปลดบล็อก (Unblacklist): ลบ Face ID ออกจาก DB เท่านั้น — ไม่ต้อง re-index ใน Rekognition เพราะ blacklist ทำงานที่ query layer
  • ผลของการปลดบล็อกมีผลทันทีในการค้นหาครั้งถัดไปของ Guest (real-time)
F-06Guest LinkP0
  • Generate unique link สำหรับ event (format: facefind.app/e/{token})
  • กำหนดอายุ link ได้ (default 7 วัน)
  • แสดงสถานะ: Active / Expired
  • Revoke link และ generate ใหม่ได้
  • Copy link ไปยัง clipboard ได้ทันที
  • Link หมดอายุ = สิทธิ์ค้นหาหมด (token ถูก revoke ที่ API) แต่รูปที่โหลดขึ้น Gallery แล้วยังดูและ download ได้ต่อจนกว่าจะปิด browser เพราะ R2 URL ไม่มีวันหมดอายุ

⚡ 8.1 Performance
  • Face search (selfie → gallery) ไม่เกิน 5 วินาที สำหรับ event 1,000 รูป
  • Gallery load web-optimized ไม่เกิน 2 วินาที (Cloudflare CDN)
  • Sync & Index รองรับ 1,000 รูปต่อ event โดยไม่ timeout — background job + batch
  • Google Drive API Rate Limit: exponential backoff, retry สูงสุด 3 ครั้งต่อ request
📈 8.2 Scale
  • Concurrent guest: เป้าหมาย 50 concurrent users ต่อ event
  • รองรับหลาย event ของหลาย tenant พร้อมกัน
  • Storage R2: ไม่มี hard limit ขยายได้ตามการใช้งาน
🔒 8.3 Security & Privacy
  • Organizer data แยก tenant ด้วย Row Level Security ใน Supabase
  • Guest session ใช้ UUID token ที่ unguessable
  • Selfie เก็บชั่วคราว ลบหลัง session expire
  • AWS Rekognition: ใบหน้าอยู่ภายใต้ AWS Data Processing Agreement
  • ต้องมี consent notice บนหน้า Guest สำหรับ PDPA compliance
🟢 8.4 Availability
  • Target uptime 99.5% (Vercel + Supabase + Cloudflare SLA)
  • Graceful error handling เมื่อ Rekognition หรือ Drive/Dropbox ไม่ตอบสนอง

Monetization Model ✅ ตัดสินใจแล้ว

โควต้าตามแผน + เก็บเงินส่วนเกิน — ลำดับการหักตอน Guest สร้าง reel:

  1. Quota — ถ้า reel_quota_used < reel_quota → หักโควต้าแผน (source = quota, used++)
  2. Add-on — โควต้าแผนหมดแต่มี reel_addon_balance > 0 → หักแพคเกจเสริม (source = addon, balance--)
  3. Per-clip — หมดทั้งคู่ → เก็บเงินต่อคลิปผ่าน payments (source = purchase + payment_id)
💡
จำนวนโควต้าต่อแผน + ราคา add-on/per-clip เป็นตัวเลขที่จะ define พร้อม plan tier (ดู Q4)
Generation Pipeline ✅ แนวทางที่แนะนำ
  • Render วิดีโอเป็นงานหนัก (CPU + เวลา) — ไม่ทำใน request เดียว ใช้ background job: Guest กดสร้าง → enqueue → render → อัพ MP4 ขึ้น R2 → set highlight_reel_status = ready
  • แนะนำ: Remotion + Lambda — เขียน reel เป็น React composition (transition / เพลง / branding คุมได้เต็ม), render serverless บน AWS (มี account อยู่แล้วจาก Rekognition), จ่ายตามจำนวน render, output ตรงขึ้น R2/S3 — เข้ากับทีม Next.js/React ที่สุด
  • Fast-start ทางเลือก: Shotstack / Creatomate — third-party render API (JSON template → MP4) ส่ง Phase 2 ได้เร็วไม่ต้องสร้าง infra เอง ค่อย migrate ไป Remotion เมื่อ volume/cost คุ้ม
  • ครอบด้วย ReelRenderer interface (แพทเทิร์นเดียวกับ PaymentService ใน §12.2) — สลับ provider ได้โดยไม่แตะ business logic · เลี่ยง self-host ffmpeg (ops หนัก) และ MediaConvert (ถนัด transcode ไม่ถนัดประกอบ montage)

12.1 Payment Model — การตัดสินใจ

✅
Phase 1 ใช้ Slip Upload + Credit System (manual verify โดย Super Admin) แทน payment gateway
  • Pay-per-event — 1 event = X credits (คงที่) · ไม่มี subscription รายเดือนใน Phase 1
  • Top-up แบบ hybrid — preset packages 3-4 ตัว + custom amount
  • โอนผ่าน PromptPay / Bank Transfer → แนบ slip → Super Admin verify → ระบบเติม credit
  • ใช้ credit activate event — event ที่ยังไม่ activate จะอยู่ใน "Draft" mode (sync/blacklist/guest link ใช้ไม่ได้)
🎯
เหตุผลที่ defer Omise: internal pilot scale ต่ำ — manual verify ไหวสบาย · ตัดเวลารอ Omise verify บริษัท 3-7 วัน · ตัดความ complex ของ webhook/subscription · ทดลอง product-market fit ก่อนผูก infra payment
🔜
Phase 2 — ย้ายไป Omise auto-topup เมื่อ scale > slip verification ไหว (ดู §11 + backlog · PaymentService abstraction รอ Phase 2 ค่อยทำ)

12.2 Credit System Architecture

3 components หลัก:

Slip Uploads
รับโอน + แนบ slip
Organizer โอนเงิน → upload slip → status pending → รอ Super Admin verify
Credit Ledger
Append-only audit trail
ทุก credit movement (topup / deduct / refund) บันทึก row ใหม่ + balance snapshot · ไม่ allow update/delete
Tenant Balance
Cached counter
Denormalized integer ใน tenants table · update atomic กับ ledger insert
Atomic Operations
  • Approve slip: update slip status + insert ledger (delta=+credits) + update tenant balance — ใน transaction เดียว
  • Activate event: check balance ≥ cost → update event activated_at + insert ledger (delta=-cost) + decrement balance — ใน transaction เดียว (กัน race condition)
  • Reject slip: update slip status + reject_reason — ไม่แตะ ledger/balance

12.4 Verification Flow (Super Admin Panel)

  1. Organizer top-up → ระบบ notify Super Admin (email / LINE / Slack)
  2. Super Admin เปิด /admin/slips → เห็นรายการ pending
  3. ตรวจ slip image vs amount + เทียบกับ statement bank → Approve หรือ Reject (ใส่ reason)
  4. Approve: ระบบทำ atomic — slip approved + ledger insert + balance +X → notify organizer
  5. Reject: notify organizer พร้อม reason → organizer แก้/upload ใหม่
⏱️
SLA: ทีมตั้ง internal target — verify ภายใน X ชม. ในเวลาทำการ (ตัวเลขให้ business ตัดสินใจ)

12.6 Phase 2 Migration Path

เมื่อ scale > slip verification ไหว — ย้ายไป Omise auto-topup:

  • เพิ่ม payments table + subscription columns ใน tenants (column nullable มีเตรียมใน §11 prep แล้ว)
  • สร้าง PaymentService interface + OmiseProvider implementation (charge/subscription/webhook)
  • เพิ่ม auto-topup logic — credit ต่ำกว่า threshold → charge Omise → topup credit อัตโนมัติ
  • UI: organizer manage payment method + view invoices · slip upload ยังคงอยู่ (fallback)
  • Existing tenants กับ credit balance ไม่ต้อง migrate — balance เดิมใช้ต่อได้, auto-topup เป็น opt-in
